🎯 Project Context

Goal 1

Prove overselling prevented for safe strategies — oversoldCount = 0

Goal 2

Prove Redis-DB consistency achievable — driftAmount = 0 for compensated strategy

Goal 3

Compare 4 strategies with measurable trade-offs — throughput, latency, oversold, drift

Goal 4

Reproducible benchmark evidence — experiment-spec.json + JMeter + saved manifests

Goal 5

Reliable event publishing — at-least-once via transactional outbox → Kafka

Non-Goals

No payment, checkout, fulfillment, buyer accounts, consumer UI, or production auth

📋 Architecture Decision Records

ADR-001: Kafka Outbox

Accepted · 2026-05-31

Transactional Outbox pattern for reliable event publishing. Events written atomically with domain changes, async published via scheduler. At-least-once delivery with retry.

ADR-002: Strategy Pattern

Accepted · 2026-04-27

Strategy + Registry pattern for stock deduction. 4 interchangeable algorithms auto-registered via @Component. Type-safe EnumMap lookup per request.

ADR-003: Redis as Gate

Accepted · 2026-04-27

MySQL is durable source of truth; Redis is fast gate/cache. Compensation + reconciliation repair drift. 2.5× throughput over DB-only with zero oversell.

⚙️ Concurrency & Consistency

Core Invariants

dbStockAfter >= 0          oversoldCount = 0          dbOrderCount <= initialStock

For REDIS_LUA_WITH_COMPENSATION: redisStockAfter = dbStockAfter, driftAmount = 0.

UNSAFE_DB is intentionally allowed to violate these — it exists to make race conditions visible.

Strategy Comparison

StrategyOversell safetyRedis drift behavior
UNSAFE_DBnon/a
CONDITIONAL_DByesRedis may differ if warmed but unused
REDIS_LUAyes (normal)Can drift on DB/order failures
REDIS_LUA_WITH_COMPENSATIONyesRestores Redis on known failures
UNSAFE_DB
DB decrement without stock predicate. Race-condition baseline — overselling is expected evidence.
UPDATE ticket_item
SET stock_available = stock_available - :qty
WHERE id = :ticketItemId
CONDITIONAL_DB
Single atomic check-and-deduct. Zero affected rows = clean rejection.
UPDATE ticket_item
SET stock_available = stock_available - :qty
WHERE id = :id AND stock_available >= :qty
REDIS_LUA
Redis Lua gate first, then DB. Fast rejection, but no compensation on failure.
1. Lua decrement Redis atomically
2. Reject if Redis < quantity
3. DB conditional update
4. No restore on failure
REDIS_LUA + COMP
Redis Lua + DB + immediate restore. Preferred strategy for demos.
1. Lua decrement Redis
2. DB conditional update
3. Restore Redis if DB rejects
4. Compensation flag for order insert fail
5. Scheduled reconcile for double-fault

Reconciliation

1. Read Redis stock     4. If drift != 0, set Redis = DB
2. Read DB stock        5. Emit flashsale.reconciliation metric
3. drift = redis - db   6. Publish RECONCILIATION event via outbox → Kafka

Scheduled every 30s (10s initial delay) for default ticket item 4. Manual: POST /admin/benchmarks/reconcile.

Failure Semantics

Result codeMeaningRecovery
INVALID_REQUESTMissing/invalid request fieldFix request
REDIS_STOCK_UNAVAILABLERedis key missing or insufficientReset + warmup
DB_STOCK_DECREMENT_FAILEDDB conditional update rejectedCheck stock, strategy, consistency
ORDER_CREATE_FAILEDStock OK but insert failedCheck consistency, reconcile
WARMUP_FAILEDCould not read DB stockVerify fixture data

Reviewer Interpretation

  • UNSAFE_DB — overselling proves the race condition.
  • CONDITIONAL_DB — simplest DB-safe baseline.
  • REDIS_LUA — fast Redis gating, drift becomes visible.
  • REDIS_LUA_WITH_COMPENSATION — best demo: fast rejection + practical repair.
  • Throughput = local evidence; rerun on target machine before publishing claims.

🔄 State Machines

OutboxEvent Lifecycle

StateEntryPurpose
PENDINGrecord() within @TransactionalAwaiting first publish attempt
PUBLISHEDKafka send OKDelivered to Kafka
FAILEDKafka error/timeoutRetryable or terminal after 5 attempts
 
PENDING → publishPendingEvents() → PUBLISHED (Kafka OK) | FAILED (error)
FAILED → retryFailedEvents() → PENDING (after 10s, attempts < 5)
FAILED → TERMINAL (attempts ≥ 5)

Stock Lifecycle

INITIAL → WARM (warmupStock) → DECREMENTING (strategy.decrease)
  → SOLD_OUT (stock=0) | OVERSOLD (stock<0, UNSAFE_DB only)
  → DRIFTED (Redis≠DB) → WARM (reconcile)

Benchmark Run

RESET → WARMED → RUNNING → COMPLETED → CHECKED → SAVED

Reconciliation

IDLE (every 30s) → CHECK (getConsistency)
  → drift=0 → IDLE
  → drift≠0 → REPAIR (setStockCache) + outbox RECONCILIATION event → IDLE

Concurrency Controls

  • Conditional UPDATE — atomic stock check-and-decrement
  • InnoDB row-level locking — implicit in UPDATE statements
  • Idempotency — in-memory cache prevents duplicate orders
  • Outbox transactional write — atomic business + event persistence
